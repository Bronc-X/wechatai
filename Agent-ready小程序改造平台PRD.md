# Agent-ready 小程序改造平台 PRD

## 1. 背景与判断

微信正在把小程序生态升级为 AI Agent 可调用的服务网络。未来用户不一定再主动打开小程序、浏览页面、填写表单，而是通过微信 AI、手机系统助手或其他 Agent 入口提出需求，再由 Agent 调用小程序完成查询、下单、预约、支付和售后。

这意味着小程序的竞争重点会从“人能不能看懂页面”扩展为“Agent 能不能稳定理解、调用、成交和恢复异常”。

本项目要做的不是通用 AI 助手，也不是简单的小程序外包开发，而是面向微信 Agent 时代的“小程序业务流 FDE”：贴近客户业务，把小程序改造成 Agent-ready 的交易与服务系统。

## 2. 产品定位

### 2.1 一句话定位

帮助小程序商家适配微信 AI / Agent 入口，让用户一句话也能完成下单、预约、诊断、支付和售后。

### 2.2 内部定位

微信生态和小程序方向的 Agent-ready FDE。

核心工作不是替客户“接入一个模型”，而是把客户的小程序页面、字段、流程、接口、数据结构、确认机制和异常处理改造成 Agent 可调用的业务系统。

### 2.3 对外表述

面向商家时，不强调 FDE 或大模型，优先使用更容易理解的话术：

> 帮你的小程序适配微信 AI 入口，让用户一句话也能完成下单、预约和支付。

## 3. 产品线设计

项目分为两条产品线：

1. Agent-ready 诊断与改造工具：全行业通用入口，用来获客、诊断、评分、生成改造建议。
2. 垂直业务流样板：聚焦高利润、高表单负担、高交易价值行业，提供深度改造方案和可复制交付包。

后续可增加第三条增强线：

3. 小程序 UI / 体验升级工具：帮助低质量小程序改善视觉、交互、表单体验和转化路径，让它既适合人使用，也适合 Agent 调用。

## 4. 目标用户

### 4.1 第一优先级客户

- 已有小程序，但转化率低、表单复杂、用户流失严重的商家。
- 有较高客单价或较高毛利的本地服务商家。
- 有预约、咨询、诊断、报价、支付、售后闭环的业务。
- 依赖微信生态获客、成交和复购的品牌或服务商。

### 4.2 典型行业

优先行业：

- 医美咨询
- 口腔 / 眼科 / 体检 / 轻医疗预约
- 汽车保养 / 维修 / 保险报价
- 家政 / 搬家 / 清洁 / 维修
- 教培 / 留学 / 职业培训咨询
- 宠物医疗 / 宠物服务
- 高端餐饮 / 茶饮连锁 / 本地生活服务

不建议第一阶段切入的行业：

- 强监管、强医疗诊断责任的正式诊疗场景。
- 交易链路过长、线下履约不可控的复杂行业。
- 客单价低、利润薄、客户无预算的小商家。

## 5. 用户问题

### 5.1 商家的问题

1. 小程序页面难看、表单复杂，用户不愿意填。
2. 商品、服务、价格、库存、号源、排班等信息没有结构化。
3. 用户需求无法直接映射到可成交商品或服务。
4. 下单、预约、支付流程太长，Agent 很难稳定完成。
5. 登录、授权、隐私、支付确认节点不清楚。
6. 失败状态不可恢复，例如无库存、时间冲突、支付失败、优惠券不可用。
7. 商家不知道未来微信 AI 入口会如何影响自己的流量和成交。

### 5.2 用户的问题

1. 不想在小程序里反复搜索、筛选、填写字段。
2. 希望用一句话表达需求，然后快速得到可选方案。
3. 下单或预约前需要清楚看到价格、时间、门店、服务内容和取消规则。
4. 出错时需要明确替代方案，而不是流程中断。

### 5.3 Agent 的问题

1. 页面元素难识别，字段含义不清楚。
2. 数据不实时，价格、库存、排班不可信。
3. 缺少标准动作，例如查询、筛选、预约、确认、支付、取消。
4. 缺少稳定的确认机制和异常恢复路径。

## 6. 成功标准

### 6.1 商业成功标准

- 能用诊断工具获得稳定商家线索。
- 能将诊断报告转化为付费改造项目。
- 每个垂直行业沉淀出可复用的字段模板、流程模板和交付包。
- 形成持续优化收入，例如月度转化分析、Agent 调用卡点分析、表单优化。

### 6.2 产品成功标准

- 10 分钟内完成一个小程序的初步 Agent-ready 评分。
- 诊断报告能清楚指出 5-10 个高优先级改造点。
- 垂直样板能演示一次完整的自然语言到成交流程。
- 改造后关键流程步骤减少，填写字段减少，失败恢复路径更清楚。

### 6.3 客户成功标准

- 小程序更容易被用户和 Agent 理解。
- 下单 / 预约 / 咨询流程更短。
- 支付前确认更安全。
- 异常流程有明确替代方案。
- 商家能看到 Agent 入口带来的转化机会。

## 7. 产品一：Agent-ready 诊断工具

### 7.1 产品目标

做一个全行业通用的小程序 Agent-ready 诊断系统，帮助商家判断自己的小程序是否适合被微信 AI / Agent 调用，并输出可执行改造建议。

### 7.2 诊断输入

MVP 阶段支持人工输入和半自动分析：

- 小程序名称 / AppID
- 行业类型
- 核心业务目标
- 关键页面截图
- 关键流程录屏或文字描述
- 商品 / 服务数据样例
- 订单 / 预约流程说明
- 当前主要转化问题

后续支持自动化：

- 页面结构抓取
- 表单字段识别
- 页面视觉质量分析
- 关键按钮和路径识别
- API / 数据结构导入
- 埋点和转化漏斗导入

### 7.3 诊断维度

#### 7.3.1 业务数据结构化

检查内容：

- 商品 / 服务是否有标准名称。
- 规格、价格、库存、门店、时间、服务条件是否机器可读。
- 是否存在大量只写在图片、富文本或人工备注中的关键信息。
- 是否支持按用户约束条件筛选。

评分标准：

- 0 分：主要信息只能靠人看页面理解。
- 3 分：部分字段结构化，但缺少规格、库存或时间。
- 5 分：核心交易字段完整结构化，Agent 可直接读取。

#### 7.3.2 用户意图映射能力

检查内容：

- 用户自然语言需求是否能映射到业务字段。
- 是否支持价格、距离、时间、偏好、禁忌、服务类型等筛选。
- 是否有同义词、口语表达和模糊需求处理。

示例：

用户说“附近 30 元以内、不太甜、15 分钟能自取的咖啡”，系统能否映射到门店、价格、糖度、库存和取餐时间。

#### 7.3.3 关键流程短链路

检查内容：

- 查询、筛选、选择、确认、支付、取消是否路径清晰。
- 是否有重复页面、无意义弹窗、强制跳转。
- 是否可以把多页操作压缩成 Agent 可执行动作。

#### 7.3.4 实时状态可信

检查内容：

- 价格是否实时。
- 库存 / 号源 / 排班是否实时。
- 门店营业状态是否实时。
- 优惠券和活动规则是否准确。

#### 7.3.5 登录与授权边界

检查内容：

- 登录是否在必要节点出现。
- 定位、手机号、联系人、健康信息等权限是否解释清楚。
- 是否存在过早授权、重复授权、授权失败后无路径的问题。

#### 7.3.6 支付前确认安全

检查内容：

- 支付前是否明确展示商品 / 服务内容。
- 是否展示价格、优惠、门店、时间、地址、取消规则。
- Agent 代操作时是否有用户确认节点。
- 是否能防止重复提交和误下单。

#### 7.3.7 异常状态恢复

检查内容：

- 无库存怎么办。
- 预约时间冲突怎么办。
- 支付失败怎么办。
- 优惠券不可用怎么办。
- 地址不在服务范围怎么办。
- 用户中途退出后能否恢复。

#### 7.3.8 售后与订单状态

检查内容：

- Agent 能否查询订单。
- 能否改约、取消、退款。
- 能否联系商家或客服。
- 订单状态是否足够清楚。

#### 7.3.9 UI / 视觉 / 表单体验

检查内容：

- 页面是否过时、粗糙或不可信。
- 信息层级是否混乱。
- 表单字段是否过多。
- 按钮是否清晰。
- 移动端是否有文本溢出、点击区域过小、加载状态缺失。
- 是否存在影响 Agent 识别的复杂浮层、图片文字、非标准控件。

这部分既服务人类用户，也服务 Agent。一个更清楚的页面，通常也更容易被 Agent 分析和调用。

### 7.4 输出内容

诊断工具输出一份报告：

- 总分
- 9 个维度分数
- 高风险问题清单
- 影响成交的问题排序
- Agent 可调用场景推荐
- 页面 / 字段 / 接口 / 流程改造建议
- UI 改造建议
- 预计改造优先级
- 建议交付包

### 7.5 MVP 功能

第一版功能：

1. 小程序信息录入。
2. 行业选择。
3. 诊断问卷。
4. 截图上传。
5. 关键流程描述。
6. 评分规则引擎。
7. AI 辅助生成诊断报告。
8. 改造建议模板。
9. PDF / Markdown 报告导出。

第一版不做：

- 不直接接入微信官方 Agent。
- 不做自动下单。
- 不做复杂爬虫。
- 不做模型微调。
- 不做通用小程序生成器。

## 8. 产品二：垂直业务流样板

### 8.1 产品目标

选择高利润、高表单负担、高交易价值行业，做可复制的 Agent-ready 改造样板。每个样板都要能演示“用户一句话 -> Agent 理解 -> 系统查询 -> 方案确认 -> 下单 / 预约 / 支付 -> 异常恢复”的完整闭环。

### 8.2 选型原则

优先选择同时满足以下条件的行业：

1. 客单价或利润较高。
2. 用户当前需要填写大量字段。
3. 交易前有咨询、诊断、预约或报价。
4. 用户需求可以被结构化。
5. 商家有转化率提升预算。
6. Agent 介入能明显减少步骤。
7. 合规风险可控。

### 8.3 第一批候选样板

#### 样板 A：医美 / 口腔咨询预约

用户需求示例：

> 我想这周末做一次牙齿清洁，预算 500 以内，离我近一点，最好能看评价。

Agent-ready 流程：

1. 解析需求：项目、预算、时间、距离、偏好。
2. 查询项目：洁牙、补牙、正畸咨询、美白等。
3. 查询门店 / 医生 / 时间。
4. 展示 2-3 个方案。
5. 用户确认后预约。
6. 支付定金或提交预约。
7. 发送预约详情。
8. 支持改约、取消、咨询客服。

关键改造点：

- 服务项目结构化。
- 医生 / 门店 / 时间 / 价格结构化。
- 适应症、禁忌、注意事项清晰化。
- 隐私授权和免责声明清晰化。
- 预约冲突和改约机制。

注意：

第一阶段只做咨询和预约，不做医疗诊断结论。

#### 样板 B：汽车保养 / 维修报价

用户需求示例：

> 我的车快 3 万公里了，帮我找附近周六能做的小保养，价格别太贵。

Agent-ready 流程：

1. 解析车型、里程、服务类型、时间、位置。
2. 查询保养项目和价格。
3. 查询门店可预约时间。
4. 给出报价和项目明细。
5. 用户确认后预约。
6. 到店前提醒。
7. 支持增项确认和售后记录。

关键改造点：

- 车型 / 里程 / 项目映射。
- 服务包结构化。
- 工时、配件、价格透明化。
- 增项确认机制。
- 门店排班和库存。

#### 样板 C：家政 / 本地上门服务

用户需求示例：

> 明天下午帮我找一个两小时的深度清洁，家里大概 90 平，预算 300 左右。

Agent-ready 流程：

1. 解析服务类型、面积、时间、地址、预算。
2. 查询服务包和服务范围。
3. 匹配可用阿姨 / 技师 / 时间段。
4. 展示价格和取消规则。
5. 用户确认后预约并支付。
6. 服务前提醒。
7. 支持改约、取消、投诉、售后。

关键改造点：

- 服务项目和计价规则结构化。
- 地址和服务半径判断。
- 人员排班。
- 取消规则。
- 售后纠纷路径。

#### 样板 D：高端餐饮 / 茶饮连锁

用户需求示例：

> 帮我找附近 30 元以内、少糖、15 分钟能自取的咖啡。

Agent-ready 流程：

1. 解析价格、口味、距离、时间。
2. 查询门店、库存、排队时间。
3. 匹配商品、规格、糖冰温。
4. 展示确认页。
5. 用户确认后支付。
6. 库存变化时给替代方案。

关键改造点：

- 菜单结构化。
- 规格和偏好结构化。
- 门店库存和排队时间。
- 优惠券可用性。
- 支付前确认。

## 9. Agent 调用规则与模型策略

### 9.1 第一阶段不优先微调模型

MVP 阶段不建议优先做大模型微调。原因：

- 数据量不足。
- 行业流程变化快。
- 微信官方能力边界尚未完全明确。
- 微调不能替代业务规则、数据结构和确认机制。

第一阶段应优先使用：

- 明确的业务 schema。
- 可配置的 Agent 调用规则。
- 工具调用 / Function Calling。
- 规则引擎。
- 少量大模型做字段抽取、意图理解、报告生成。

### 9.2 什么时候考虑微调

满足以下条件后再考虑微调：

- 已有多个同行业客户。
- 收集到大量真实需求表达。
- 已有足够多的成功 / 失败调用日志。
- 发现通用模型在特定行业术语、字段抽取、意图分类上稳定出错。

可微调方向：

- 行业意图分类。
- 表单字段自动填充。
- 用户需求到服务参数映射。
- 异常原因归类。
- 诊断报告生成风格。

### 9.3 推荐技术路线

第一阶段：

- 规则优先。
- Schema 优先。
- 模型辅助。
- 人工确认关键动作。

第二阶段：

- 建立行业动作库。
- 建立模型网关。
- 建立调用日志和转化分析。
- 对高频任务做轻量模型优化。

第三阶段：

- 针对行业做专用模型或微调。
- 做 Agent 调用模拟和自动回归测试。
- 做跨客户的行业基准评分。

## 10. 技术架构

### 10.1 总体架构

系统分为五层：

1. 微信生态层：小程序、微信支付、微信登录、微信 AI / A2A、官方审核。
2. 前端体验层：小程序页面、表单、确认页、状态反馈、UI 改造组件。
3. 业务适配层：商品 / 服务 schema、订单、预约、库存、排班、售后。
4. Agent 编排层：意图解析、参数映射、动作选择、确认机制、异常恢复。
5. 分析与优化层：诊断评分、调用日志、漏斗分析、报告生成、UI 评分。

### 10.2 你真正要控制的层

你不控制微信官方 Agent 入口，也不控制手机厂商 A2A 入口。

你主要控制：

- 小程序前端改造。
- 表单和确认页设计。
- 商品 / 服务字段结构。
- 业务接口适配。
- Agent 调用规则。
- 异常恢复机制。
- 转化和失败日志。
- 诊断与优化报告。

所以这个项目表面上是“前端部署和前端调用”，但真正护城河在“业务流适配层”。

### 10.3 部署建议

MVP 阶段：

- 优先腾讯云 CloudBase / 微信云托管。
- 使用云函数或轻量后端服务。
- 使用数据库保存诊断项目、评分、报告和模板。
- 大模型通过 API 调用，不自建 GPU。

客户项目阶段：

- 客户小程序前端和业务后端按实际情况接入。
- 你的平台提供诊断、规则配置、报告、日志分析。
- 对敏感业务数据做最小化采集。

规模化阶段：

- 增加模型网关，统一接入混元、DeepSeek、Kimi、GLM、MiniMax 等模型。
- 增加行业模板市场。
- 增加 Agent 调用回归测试系统。

## 11. UI / 小程序美化工具

### 11.1 为什么要做

很多小程序页面丑、旧、乱，不只是审美问题，还会影响信任感、转化率和 Agent 识别稳定性。

未来小程序体验优化应同时服务两类用户：

1. 人类用户：看得懂、愿意点、信任页面。
2. Agent：能识别页面结构、字段含义、动作路径和状态反馈。

### 11.2 产品形态

UI / 体验工具可以作为 Agent-ready 诊断工具的一个模块，不建议第一阶段独立做成大而全的小程序生成器。

功能包括：

- 页面视觉评分。
- 信息层级检查。
- 表单字段冗余检查。
- 按钮和 CTA 检查。
- 移动端适配检查。
- 文本溢出和点击区域检查。
- 加载 / 空状态 / 错误状态检查。
- 页面改版建议。
- 关键页面重设计稿或组件建议。

### 11.3 交付方式

第一阶段交付：

- 页面问题清单。
- 改版建议。
- 关键页面 wireframe。
- 小程序组件规范。
- 表单瘦身方案。

第二阶段交付：

- 生成式页面改版草稿。
- 小程序组件库。
- 行业页面模板。
- 设计到代码辅助工具。

## 12. MVP 范围

### 12.1 MVP 目标

在 4-6 周内做出一个可演示、可销售的 Agent-ready 诊断工具和一个垂直样板。

### 12.2 MVP 包含

1. Agent-ready 评分问卷。
2. 截图上传。
3. 诊断报告生成。
4. UI / 表单体验评分。
5. 行业模板：医美 / 口腔预约或高端本地服务。
6. 一个完整自然语言交易演示脚本。
7. 一份客户交付版报告模板。
8. 一套改造报价包。

### 12.3 MVP 不包含

1. 不承诺直接接入微信官方 AI。
2. 不做真实支付自动化。
3. 不碰强医疗诊断结论。
4. 不做自研大模型。
5. 不做全自动小程序生成。

## 13. 核心用户流程

### 13.1 诊断工具流程

1. 商家提交小程序信息。
2. 选择行业和核心目标。
3. 上传关键页面截图。
4. 描述一个核心业务流程。
5. 系统生成初步评分。
6. 系统输出问题清单。
7. 系统推荐 Agent-ready 改造路径。
8. 销售或顾问跟进付费改造。

### 13.2 垂直改造流程

1. 访谈商家业务流程。
2. 梳理商品 / 服务 / 价格 / 预约 / 支付 / 售后字段。
3. 定义 Agent 可调用动作。
4. 设计支付前确认和失败恢复路径。
5. 改造小程序关键页面和接口。
6. 做模拟 Agent 调用测试。
7. 上线后持续分析卡点和转化。

### 13.3 UI 优化流程

1. 上传页面截图。
2. 系统识别视觉、层级、表单和交互问题。
3. 输出页面优化建议。
4. 生成改版 wireframe。
5. 开发落地关键页面。
6. 对比改版前后转化数据。

## 14. 商业模式

### 14.1 免费 / 低价入口

- Agent-ready 初步评分。
- 小程序体验体检。
- 行业趋势报告。

目标：获客和建立认知。

### 14.2 标准化诊断包

交付：

- 完整诊断报告。
- 改造优先级。
- 业务流程图。
- 字段和接口建议。
- UI / 表单优化建议。

适合：中小商家、代理商、服务商。

### 14.3 垂直行业改造包

交付：

- 行业 schema。
- 关键页面改造。
- 预约 / 下单 / 支付确认流程。
- 异常恢复机制。
- 模拟 Agent 调用脚本。
- 上线验证。

适合：高客单商家、连锁品牌、服务商。

### 14.4 持续优化订阅

交付：

- 月度 Agent-ready 评分。
- 转化漏斗分析。
- 失败原因分析。
- 表单优化建议。
- 新页面审查。
- 行业 benchmark。

这是长期更有价值的收入层。

## 15. 关键指标

### 15.1 获客指标

- 诊断工具访问量。
- 完成诊断数量。
- 报告下载数量。
- 诊断到咨询转化率。

### 15.2 交付指标

- 改造项目成交率。
- 平均客单价。
- 交付周期。
- 客户复购率。

### 15.3 业务结果指标

- 表单字段减少比例。
- 核心流程步骤减少比例。
- 预约 / 下单转化率提升。
- 支付前流失率下降。
- 异常恢复成功率。
- 客服人工介入率下降。

### 15.4 Agent-ready 指标

- 意图识别成功率。
- 参数映射成功率。
- 可执行动作覆盖率。
- 支付前确认完整率。
- 失败恢复路径覆盖率。

## 16. 风险与应对

### 16.1 微信官方替代风险

风险：

微信可能推出官方自动接入、官方诊断、官方开发者工具。

应对：

- 不做通用接入层护城河。
- 深入垂直行业流程。
- 沉淀行业 schema、模板、日志和改造方法。
- 把官方能力当渠道和基础设施，而不是竞争对手。

### 16.2 模型能力不稳定

风险：

大模型对业务字段理解不稳定，容易误下单或错填。

应对：

- 规则和 schema 优先。
- 关键动作必须用户确认。
- 支付动作不由模型直接决定。
- 保留人工兜底和异常恢复。

### 16.3 合规风险

风险：

医疗、金融、隐私、支付场景存在监管和责任问题。

应对：

- 第一阶段只做咨询、预约、导购，不做诊断结论。
- 敏感字段最小化采集。
- 明确用户授权。
- 关键决策给用户确认。

### 16.4 交付过重

风险：

FDE 模式容易变成定制外包。

应对：

- 每个项目必须沉淀模板。
- 行业字段、流程、页面、报告标准化。
- 三个客户后抽象成行业包。
- 限制非核心定制。

## 17. 路线图

### 阶段 0：准备期，1-2 周

- 完成诊断评分表。
- 完成报告模板。
- 完成 2 个行业样板设计。
- 准备销售话术和演示脚本。

### 阶段 1：MVP，4-6 周

- 做出诊断工具原型。
- 支持截图上传和问卷评分。
- 生成 Markdown / PDF 报告。
- 做出一个垂直样板演示。
- 找 3-5 个真实小程序试诊断。

### 阶段 2：试点客户，6-10 周

- 交付 1-2 个付费改造项目。
- 沉淀行业 schema。
- 沉淀小程序 UI 改造组件。
- 建立调用日志和转化分析框架。

### 阶段 3：产品化，3-6 个月

- 做成标准 SaaS 工具。
- 增加行业模板库。
- 增加模型网关。
- 增加 Agent 调用模拟测试。
- 建立服务商合作模式。

2. 小程序体验和 UI 评分表。
3. 诊断报告模板。

## 19. 当前最推荐的下一步

建议先做两个并行资产：

1. 全行业诊断工具原型。
2. 一个高利润、高表单负担行业的垂直样板。

垂直样板建议优先在“医美 / 口腔咨询预约”和“家政 / 本地上门服务”中选一个。它们比普通餐饮更有利润空间，也更能体现 Agent 减少字段填写、提升预约转化的价值。

餐饮 / 咖啡样板适合做公开演示，因为风险低、用户容易理解，但未必是第一批商业化最强的切入口。

## 20. 核心结论

这个项目不是做一个“AI 小程序工具”，而是做微信 Agent 时代的小程序交易基础设施改造。

工具线负责规模化获客和标准化诊断。

垂直业务流负责高价值交付和行业壁垒。

UI / 体验升级负责提升人类用户转化，同时让页面更适合 Agent 理解。

真正的护城河不在某个大模型，而在行业流程、业务 schema、转化日志、失败恢复机制和贴近客户业务的交付能力。
