# 垂直样板：医美 / 口腔咨询预约

## 样板目标

搭建一个高利润、高表单负担、合规风险可控的 Agent-ready 业务流样板。

第一版只覆盖咨询和预约，不做医疗诊断结论。

## 目标客户

- 口腔门诊。
- 轻医美机构。
- 体检中心。
- 眼科 / 皮肤科咨询预约类小程序。

## 典型用户需求

用户可能这样表达：

- “我这周末想洗牙，预算 500 以内，离我近一点。”
- “想咨询一下牙齿美白，最好晚上能去。”
- “我想做皮肤检测，周六有没有空位？”
- “帮我找一个评价好一点的口腔门诊，先预约咨询。”

## Agent 需要解析的参数

| 参数 | 示例 | 是否必须 |
| --- | --- | --- |
| 服务项目 | 洁牙、牙齿美白、正畸咨询、皮肤检测 | 是 |
| 时间偏好 | 本周末、晚上、明天下午 | 是 |
| 位置偏好 | 附近、某商圈、某区 | 否 |
| 预算 | 500 以内 | 否 |
| 门店偏好 | 评价好、离地铁近 | 否 |
| 医生偏好 | 有经验、女性医生 | 否 |
| 用户信息 | 姓名、手机号 | 预约前必须 |
| 特殊情况 | 孕期、过敏、疾病史 | 视项目而定 |

## 标准业务流

1. 用户提出自然语言需求。
2. Agent 提取服务项目、时间、位置、预算等参数。
3. 系统查询服务项目和可预约门店。
4. 系统查询医生 / 排班 / 号源。
5. 系统生成 2-3 个候选方案。
6. 用户选择方案。
7. 系统展示预约前确认页。
8. 用户确认后提交预约或支付定金。
9. 系统生成预约订单。
10. 用户可改约、取消或联系客服。

## 支付前 / 预约前确认字段

确认页必须展示：

- 服务项目。
- 门店名称和地址。
- 预约时间。
- 医生 / 咨询师信息。
- 价格或定金。
- 到店注意事项。
- 取消 / 改约规则。
- 隐私与健康信息授权说明。
- 是否只是咨询预约，不构成医疗诊断。

## 关键页面改造

### 项目列表页

需要结构化：

- 项目名称。
- 项目分类。
- 价格区间。
- 适合人群。
- 不适合人群。
- 预计耗时。
- 是否需要到店。
- 是否支持线上咨询。

### 门店 / 医生页

需要结构化：

- 门店地址。
- 营业时间。
- 距离。
- 可预约时间。
- 医生资质。
- 擅长项目。
- 用户评价。

### 预约表单页

优化目标：

- 减少用户手填字段。
- 可由 Agent 预填项目、时间、门店、预算。
- 手机号和隐私授权在提交前明确确认。
- 敏感信息只在必要时收集。

### 预约确认页

优化目标：

- 让用户在最后一步确认所有关键信息。
- 防止 Agent 误约、重复提交或错选项目。
- 清楚展示取消和改约规则。

## 异常恢复路径

| 异常 | 恢复方案 |
| --- | --- |
| 目标时间无号 | 推荐相邻时间或其他门店 |
| 预算不匹配 | 推荐更低价项目或咨询套餐 |
| 门店太远 | 推荐距离更近的门店 |
| 项目不适合 | 提示先做咨询，不直接预约项目 |
| 支付失败 | 保留预约信息，引导重新支付 |
| 用户信息缺失 | 只补充缺失字段，不重填全部表单 |
| 用户中途退出 | 保存草稿，返回后继续 |

## 合规边界

第一版必须遵守：

- 不输出医疗诊断结论。
- 不替医生判断治疗方案。
- 不承诺效果。
- 不处理高风险医疗决策。
- 只做咨询、预约、信息整理和用户确认。

## Demo 脚本

用户：

> 我这周末想洗牙，预算 500 以内，离我近一点。

Agent 解析：

- 项目：洁牙。
- 时间：本周末。
- 预算：500 元以内。
- 位置：附近。
- 目标：预约咨询 / 到店服务。

系统返回：

1. A 门店，周六 15:00，洁牙套餐 398 元，距离 1.2km。
2. B 门店，周日 10:30，洁牙套餐 468 元，距离 2.1km。
3. C 门店，周六 18:00，洁牙套餐 299 元，距离 3.0km。

Agent：

> 你想预约 A 门店周六 15:00 的 398 元洁牙套餐吗？预约前需要确认手机号，并提示这只是洁牙服务预约，不构成医疗诊断。

用户确认后：

- 生成预约。
- 发送门店地址、时间、价格、取消规则。
