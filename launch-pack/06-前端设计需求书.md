# 前端设计需求书

## 1. 产品定位

本产品是面向小程序商家、买卖家和商家服务商的 Agent-ready 小程序改造交付包工具。

它不是通用小程序生成器，也不是单纯的 UI 美化工具，而是帮助服务团队把一个小程序项目整理成可诊断、可评分、可行动、可交付、可导出的改造工作台。

核心价值：

1. 用项目分类管理不同客户或不同行业的小程序改造项目。
2. 用交付封面和项目档案沉淀客户背景、行业、目标流程和交付范围。
3. 用诊断评分发现小程序在 Agent-ready、表单体验、流程结构、确认机制和异常恢复上的问题。
4. 用行动清单把诊断结论转成可执行改造任务。
5. 用证据附录和报告导出支持客户沟通、报价、复盘和交付验收。

设计上应呈现为一个清晰、可信、轻量但有高级感的工作台。它需要让用户感到“这是一套可以拿去和客户开会的专业交付系统”，而不是一个临时表单或普通后台。

## 2. 目标用户

### 2.1 主要用户

| 用户 | 需求 | 设计重点 |
| --- | --- | --- |
| 小程序改造服务商 | 快速管理客户项目、输出诊断和报告 | 项目视图清楚、导出路径明确、任务推进感强 |
| 商家运营负责人 | 理解小程序为什么需要改造、哪些问题优先修 | 评分解释直观、行动建议可读、风险和收益清楚 |
| 买卖家 / 客户沟通角色 | 查看交付范围、确认需求、追踪改造进展 | 信息层级简单、状态明确、术语不过重 |
| 前端 / 产品交付团队 | 按诊断结果拆任务、收集证据、形成交付文档 | 组件化、状态完整、证据和任务可追溯 |

### 2.2 使用场景

1. 服务商新建一个客户项目，录入小程序名称、行业、核心目标和业务流程。
2. 咨询或诊断过程中，按维度给小程序评分并记录证据。
3. 系统生成行动清单，服务商调整优先级并分配给交付团队。
4. 项目完成阶段，整理证据附录并导出诊断或交付报告。
5. 商家查看报告，理解改造范围、优先级和后续报价。

## 3. 信息架构

建议采用二段式或三段式布局，优先保证清晰扫描和持续工作效率。

### 3.1 全局结构

推荐三段式：

1. 左侧项目导航：项目分类、项目列表、状态筛选。
2. 中央主工作区：当前功能页面的核心内容。
3. 右侧上下文面板：项目摘要、评分概览、当前报告状态、快捷导出。

在屏幕宽度不足时降级为二段式：

1. 顶部或抽屉式项目导航。
2. 主工作区，右侧上下文面板折叠为底部面板或浮层。

### 3.2 功能分类

产品按项目分类管理四个核心功能：

| 功能 | 页面目标 | 关键内容 |
| --- | --- | --- |
| 交付封面 / 项目档案 | 建立项目上下文和客户交付边界 | 项目名称、行业、客户、目标流程、交付状态、负责人、时间线 |
| 诊断评分 | 评估小程序 Agent-ready 程度 | 评分维度、分数、证据、风险等级、改造建议 |
| 行动清单 | 把诊断转为可执行任务 | 任务标题、优先级、负责人、状态、关联证据、预计收益 |
| 证据附录与报告导出 | 汇总证据并生成可交付报告 | 截图、链接、备注、引用关系、导出格式和报告预览 |

### 3.3 推荐页面流

1. 项目首页：查看所有项目、分类、状态和最近更新。
2. 项目详情：进入单个项目后默认显示交付封面和项目档案。
3. 诊断评分：完成评分、记录证据、查看维度解释。
4. 行动清单：按优先级推进改造任务。
5. 证据附录：集中管理截图、页面链接、问题说明和验证结果。
6. 报告导出：预览并导出 Markdown、PDF 或前端后续支持的报告格式。

## 4. 页面与组件要求

### 4.1 项目首页

目标：让用户快速找到项目，并理解每个项目处于什么状态。

组件要求：

1. 项目分类导航：支持按行业、客户类型、交付状态筛选。
2. 项目列表：每个项目展示名称、行业、状态、总分、最近更新时间、负责人。
3. 新建项目入口：清楚但不喧宾夺主。
4. 状态概览：展示待诊断、诊断中、待交付、已完成数量。
5. 搜索与筛选：支持按项目名、行业、客户名搜索。

视觉要求：

- 项目卡片可以使用半透明玻璃层，但信息密度要适合工作台，不要做成营销页大卡片。
- 状态标签要清楚区分，例如未开始、诊断中、行动中、待导出、已交付。

### 4.2 交付封面 / 项目档案页

目标：形成一个可交给客户看的项目首页。

组件要求：

1. 项目封面区：项目名称、客户、行业、核心目标、当前阶段。
2. 项目档案表单：客户背景、业务流程、目标用户、关键页面、交付范围。
3. 时间线：记录诊断、建议、改造、验证、报告导出节点。
4. 项目健康摘要：总分、最高风险维度、下一步行动。
5. 快捷操作：进入评分、生成行动清单、导出报告。

设计要求：

- 封面区应有明显层级感，可使用柔和液态玻璃效果表现“交付包封面”。
- 表单应紧凑、易扫读，避免大面积空洞留白。
- 编辑态和只读态要区分清楚。

### 4.3 诊断评分页

目标：支持用户高效完成评分，并把每个分数和证据绑定。

组件要求：

1. 评分维度列表：展示维度名称、分数、等级、简短解释。
2. 评分控件：建议使用 0-5 或 0-10 分的滑块、分段按钮或星级式控件。
3. 证据输入：每个维度可关联截图、页面链接、备注和问题描述。
4. 风险提示：低分项自动突出显示，并提示可能影响的业务结果。
5. 总分面板：展示总分、等级、改造优先级和建议摘要。

交互要求：

- 修改分数后，总分和等级应即时更新。
- 低分维度应通过颜色、图标和微动效提醒，但不要过度警示。
- 支持展开 / 收起维度详情，减少页面拥挤。

### 4.4 行动清单页

目标：把诊断结果转成可执行、可追踪的改造任务。

组件要求：

1. 任务列表：标题、优先级、状态、负责人、关联评分维度、预计收益。
2. 任务详情面板：问题背景、建议动作、验收标准、证据链接。
3. 状态切换：待处理、进行中、待验证、已完成。
4. 优先级排序：高、中、低，或按影响 / 成本矩阵排序。
5. 批量操作：支持选择多个任务进行状态更新或加入报告。

设计要求：

- 行动清单应像轻量项目管理工具，而不是普通表格。
- 任务优先级需要一眼能看懂。
- 已完成任务要有低干扰的完成反馈。

### 4.5 证据附录与报告导出页

目标：把诊断证据和行动建议整理成可交付材料。

组件要求：

1. 证据库：支持截图、页面链接、文本备注、评分维度引用。
2. 证据卡片：展示缩略图、标题、来源页面、关联任务或评分项。
3. 报告结构预览：封面、项目档案、评分结果、行动清单、证据附录。
4. 导出设置：选择报告类型、包含内容、格式和语言。
5. 导出状态：生成中、成功、失败、可重新导出。

设计要求：

- 报告预览要有纸面感和正式交付感。
- 导出按钮必须有明确状态反馈，避免用户重复点击。
- 证据和报告之间的引用关系要可追溯。

## 5. 交互动效要求

技术栈必须使用 Motion 动效。可选 React + Motion 或 Tailwind + Motion 组合。

动效方向参考 iOS 26 的 Glass Liquid 视觉趋势，但不得照抄系统 UI、商标、图标、控件或专有视觉元素。应抽象为以下设计语言：

1. 液态玻璃：半透明、柔和折射、轻微模糊、边缘高光。
2. 层级光泽：关键面板有细腻光泽层次，帮助区分前后景。
3. 柔和动态：状态切换和面板展开像轻微流动，不做夸张弹跳。
4. 可信工作台：动效服务于理解、状态反馈和空间变化，不做炫技。

### 5.1 必需动效

1. 页面切换：项目首页、详情页、评分页、行动清单、报告页之间使用轻量淡入和位移。
2. 面板展开：评分维度、任务详情、证据详情使用高度或透明度过渡。
3. 数字变化：总分、维度分、项目数量变化时使用平滑递增或淡入。
4. 状态反馈：保存成功、导出成功、任务完成需要有短暂反馈。
5. 悬停反馈：按钮、项目卡片、证据卡片使用轻微光泽、边框或阴影变化。
6. 加载状态：导出报告、生成建议、保存项目时使用柔和 loading，而不是突然空白。

### 5.2 动效限制

1. 避免大面积背景持续运动，防止干扰工作。
2. 避免过强弹性、抖动、旋转和夸张缩放。
3. 动效时长建议控制在 120-280ms，页面级过渡不超过 400ms。
4. 必须支持减少动态效果设置，用户开启 reduced motion 时禁用非必要位移和缩放。

## 6. 视觉规范

### 6.1 总体气质

关键词：

- 简洁清晰
- 专业可信
- 液态玻璃
- 半透明层级
- 柔和动态
- 交付包质感

整体应介于专业 SaaS 工作台和高质感客户交付文档之间。不要做成营销落地页，也不要做成传统重后台。

### 6.2 色彩

建议使用冷静中性色作为底色，搭配少量功能色：

| 用途 | 建议 |
| --- | --- |
| 背景 | 低饱和浅灰、雾白、淡蓝灰或深浅可切换中性色 |
| 主色 | 清爽蓝、青蓝或蓝绿色，用于主按钮和关键状态 |
| 成功 | 柔和绿色 |
| 警示 | 温和琥珀色 |
| 风险 | 克制红色，仅用于高风险或错误 |
| 玻璃层 | 半透明白或深色透明层，配合模糊和细边框 |

避免：

- 大面积紫蓝渐变。
- 过度霓虹色。
- 纯装饰性光球、模糊色块或无意义背景动效。
- 与 iOS 系统界面高度相似的配色、图标和控件组合。

### 6.3 字体与排版

1. 中文优先使用系统字体栈，确保 Windows、macOS 和移动端清晰。
2. 页面标题清楚克制，不使用过大的英雄标题。
3. 表格、任务和评分区域使用紧凑排版，便于长时间工作。
4. 数字评分可以使用更强的字重和尺寸，形成清晰焦点。
5. 文案不应挤压按钮和标签，所有文本必须在移动端可换行或截断。

### 6.4 组件风格

1. 面板：半透明玻璃层、细边框、轻阴影，圆角控制在 8-16px。
2. 按钮：主按钮清楚，次按钮克制，图标按钮需要 tooltip 或可访问标签。
3. 表单：输入框边界清楚，聚焦态明显，错误提示贴近字段。
4. 标签：状态和优先级使用颜色加文字，不只依赖颜色。
5. 卡片：只用于项目、任务、证据等重复项目，不嵌套卡片。
6. 表格：适合高密度信息；在移动端可转换为列表。

## 7. 响应式要求

### 7.1 桌面端

目标宽度：1280px 以上优先。

要求：

1. 默认三段式布局。
2. 左侧项目导航固定或可折叠。
3. 中央内容区保持最强视觉权重。
4. 右侧上下文面板可固定展示项目摘要和快捷操作。
5. 表格和清单需要支持横向空间充分利用。

### 7.2 平板端

目标宽度：768-1279px。

要求：

1. 降级为二段式布局。
2. 右侧上下文面板变为可打开抽屉。
3. 项目导航可折叠。
4. 评分维度和任务列表保持可读，不应出现横向滚动依赖。

### 7.3 移动端

目标宽度：360-767px。

要求：

1. 使用单列布局。
2. 项目导航放入顶部菜单或底部切换入口。
3. 评分、任务、证据以列表卡片呈现。
4. 导出和保存等关键操作固定在可见区域，但不能遮挡内容。
5. 所有表单字段、按钮和标签必须避免文字溢出。

## 8. 可访问性要求

1. 所有可点击元素必须支持键盘访问。
2. 焦点态必须可见，不能只靠浏览器默认细线。
3. 状态、风险、优先级不能只靠颜色表达，必须配合文字或图标。
4. 表单字段必须有明确 label、错误提示和帮助说明。
5. 图片证据必须支持替代文本或备注说明。
6. 动效必须支持 reduced motion。
7. 文字和背景对比度需要满足常规可读要求。
8. 导出、保存、生成等异步状态需要被屏幕阅读器感知。

## 9. 交付清单

前端设计团队需要交付以下内容：

1. 信息架构图：项目首页、项目详情、评分、行动清单、证据附录、报告导出的页面关系。
2. 桌面端高保真稿：至少覆盖 1280px 桌面视口。
3. 移动端高保真稿：至少覆盖 390px 移动视口。
4. 核心组件库：导航、项目卡片、评分控件、任务项、证据卡片、报告预览、导出按钮、状态标签、表单字段、空状态、错误状态、加载状态。
5. Motion 动效说明：页面切换、面板展开、数字变化、保存反馈、导出反馈、卡片悬停。
6. 视觉规范：颜色、字体、间距、圆角、阴影、玻璃材质、图标使用规则。
7. 响应式规则：桌面、平板、移动端布局变化。
8. 可访问性说明：键盘、焦点、对比度、reduced motion、表单错误提示。
9. 设计交付标注：关键间距、组件状态、交互说明、异常状态。

## 10. 验收标准

### 10.1 产品完整性

1. 设计稿完整覆盖四个核心功能：交付封面 / 项目档案、诊断评分、行动清单、证据附录与报告导出。
2. 项目分类管理在首页和项目详情中都能清楚体现。
3. 用户能从新建项目一路走到评分、生成行动、整理证据和导出报告。
4. 每个关键页面都有 loaded、loading、empty、error 状态。

### 10.2 设计质量

1. 布局简洁清晰，桌面端优先使用二段式或三段式结构。
2. 液态玻璃方向体现在层级、半透明、光泽和柔和动态上，但不照抄任何系统 UI 或商标元素。
3. 页面气质专业可信，适合服务商拿给客户演示。
4. 信息密度适合工作台，不出现营销页式的大段宣传内容。
5. 卡片、面板、表单、表格和报告预览风格统一。

### 10.3 交互与动效

1. Motion 动效覆盖页面切换、面板展开、数字变化、状态反馈和悬停反馈。
2. 动效节制、柔和，不影响阅读和录入。
3. reduced motion 开启时非必要动态可以关闭或明显减弱。
4. 保存、导出、生成建议等异步行为都有明确反馈。

### 10.4 响应式与可访问性

1. 1280px 桌面、768px 平板、390px 移动端均无明显内容溢出。
2. 移动端可以完成项目查看、评分查看、任务查看和报告导出入口操作。
3. 键盘可访问、焦点可见、对比度可读。
4. 状态和风险信息不只依赖颜色。

### 10.5 技术落地

1. 设计适合 React 或 Tailwind 技术栈实现。
2. 组件状态清楚，可拆分为前端可复用组件。
3. Motion 动效说明足够明确，前端可以直接按页面和组件实现。
4. 不依赖无法落地的复杂视觉特效；玻璃效果应能通过透明度、模糊、边框、高光和阴影实现。
